# 🔍 Database Schema Critique: LaaS Enterprise Database Design

## Executive Summary

The database design in `laas_enterprise_database_design_1560fd47.plan.md` is **architecturally solid and comprehensive** for a 60+ table enterprise system. However, I've identified **critical gaps**, **inconsistencies with the project spec**, and **technical concerns** that need addressing before implementation.

---

## 🚨 CRITICAL ISSUES (Must Fix)

### 1. **Storage Quota Mismatch — 5GB vs 15GB**

| Location | Value | Issue |
|----------|-------|-------|
| `user_storage_volumes.quota_bytes` | `16106127360` (≈15 GiB) | ✅ Correct |
| Current Prisma `users.storageUid` docs | `5GB` (5GiB = 5368709120) | ❌ **SPEC MISMATCH** |
| `org_resource_quotas.max_storage_per_user_mb` | `15360` (15GB) | ✅ Correct |

**Action Required**: Update documentation and ensure storage provisioning service uses 15GB (16106127360 bytes), not 5GB.

---

### 2. **Missing GPU Isolation Fields in Sessions**

The spec explicitly requires **dual-layer VRAM enforcement** (HAMi-core + CUDA MPS), but the `sessions` table lacks:

```prisma
// MISSING - Required for GPU session management
gpu_fraction_sm_percent      Int?    // HAMi SM allocation (e.g., 25 for 25%)
mps_active_percent           Int?    // CUDA MPS compute limit
hami_container_name          String? // HAMi container for VRAM tracking
gpu_utilization_snapshot     JSONB?  // Periodic GPU stats from nvidia-smi
vram_snapshot_bytes          BIGINT? // VRAM used at snapshot time
```

**Impact**: No way to track GPU resource consumption, verify HAMi enforcement, or generate GPU billing reports.

---

### 3. **Public Users Can Access Stateful Sessions**

The `compute_config_access` table design is correct, but there's **no enforcement at the booking layer**:

- `bookings` table allows any user to create a booking
- No check preventing `public_user` role from booking `stateful_desktop` configs
- Need a **DB-level constraint or application guard** that validates:
  ```sql
  -- Pseudo-code for required check
  IF user_role = 'public_user' AND session_type = 'stateful_desktop'
    THEN REJECT booking
  ```

---

### 4. **Session Booking Conflict Detection Missing**

The spec requires **concurrent sessions per node** based on resource availability, but:

| Missing Element | Purpose |
|-----------------|---------|
| `node_id` in `bookings` | Which node the slot reserves |
| Resource validation on booking | Check `allocated_vcpu + booking_vcpu <= total_vcpu` |
| No `booking_locks` table | Prevent double-booking same slot |
| No waitlist mechanism | When resources are exhausted |

**The `bookings` table needs `node_id` and validation logic before Phase 1 launch.**

---

### 5. **No Wallet Balance Check Before Session Launch**

The spec requires **pre-authorization** before launching a session:

```prisma
// MISSING - Required for billing protection
booking_preauth_transaction_id UUID? // Wallet hold at booking time
minimum_balance_required_cents BIGINT? // Calculated at booking
```

Without this, users can book sessions and drain their wallet mid-session.

## ⚠️ SIGNIFICANT CONCERNS (Should Fix)

### 6. **Race Condition: Concurrent OTP Verification**

The current `otp_verifications` pattern in the auth service:
```typescript
// Line 100-106: Race condition window exists
const record = await this.prisma.otpVerification.findFirst({...});
// Another request could use the same OTP here
if (!record || record.expiresAt < new Date()) {...}
```

**Fix**: Use `UPDATE ... WHERE attempts < 5` with `RETURNING *` or row-level locking:
```typescript
const record = await this.prisma.$queryRaw`
  SELECT * FROM otp_verifications 
  WHERE email = ${email} AND used_at IS NULL 
  ORDER BY created_at DESC LIMIT 1 FOR UPDATE
`;
```

---

### 7. **Soft Delete Not Applied Consistently**

The plan specifies soft delete on 40+ tables, but it's **NOT applied to critical operational tables**:

| Table | Has Soft Delete | Should Have? |
|-------|----------------|--------------|
| `sessions` | ❌ No | ❌ No (append-only makes sense) |
| `bookings` | ❌ No | ⚠️ Debatable |
| `wallet_transactions` | ❌ No | ✅ Correct (append-only) |
| `billing_charges` | ❌ No | ✅ Correct (append-only) |
| `nodes` | ❌ No | ⚠️ **YES** - Need to preserve historical node data |

**`nodes` should have `deleted_at`** to preserve:
- Historical session data referencing the node
- Audit trail of node lifecycle
- GPU utilization history even after decommissioning

### 8. **Missing Ephemeral vs Stateful Storage Differentiation**

The spec distinguishes storage behavior:

| User Type | Session Type | Storage Behavior |
|-----------|--------------|------------------|
| University member | Stateful | 15GB persistent ZFS volume |
| University member | Ephemeral | Wiped after session, notebooks saved via `user_files` |
| Public user | Ephemeral | **ONLY ephemeral** - notebooks saved to `user_files` |

**Missing**: `sessions.is_storage_persistent` flag to differentiate:
```prisma
is_storage_persistent  BOOLEAN DEFAULT false  -- True for stateful, false for ephemeral
storage_mount_type      VARCHAR(32)             -- 'nfs_persistent' | 'tmpfs_ephemeral'
```

---

### 9. **No Session Reconnection Tracking**

The spec requires **session reconnection**:
> "If browser closes mid-session: container continues running... Reconnect button → WebRTC reconnects"

**Missing in `sessions` table**:
```prisma
reconnect_count          INT DEFAULT 0         -- Number of reconnections
last_reconnected_at      TIMESTAMPTZ?           -- Last reconnect time
webtrc_session_token     String?                -- For re-authenticating reconnects
```

**Missing table**: `session_connections` for detailed connection history:
```prisma
model session_connections {
  id            UUID PK
  session_id    UUID FK
  user_id       UUID FK
  client_ip     INET
  user_agent    TEXT
  connected_at  TIMESTAMPTZ
  disconnected_at TIMESTAMPTZ
  disconnect_reason VARCHAR(32)  -- 'user_close' | 'network' | 'timeout'
}
```

### 13. **No Resource Utilization Tracking Table**

The spec requires **audit logging of resource usage**:
> "Resource utilization: CPU%, RAM%, GPU%, VRAM peak, disk written, network transferred"

**The `sessions.resource_snapshot` JSONB is insufficient** for:
- Historical analysis (capacity planning)
- Billing disputes
- Performance optimization

**Add table**:
```prisma
model session_resource_samples {
  id            UUID PK
  session_id    UUID FK
  sampled_at    TIMESTAMPTZ NOT NULL
  cpu_percent   DECIMAL(5,2)
  memory_used_mb INT
  gpu_util_percent DECIMAL(5,2)
  vram_used_mb  INT
  disk_io_bytes BIGINT
  network_bytes BIGINT
  container_cpu_seconds DECIMAL(10,2)  -- For billing calculation
}
-- Partition by session_id with TTL of 90 days
-- Hot data in MongoDB, summarized to sessions.resource_snapshot at end
```

# Smart Storage Upgrade with Cross-Node Migration

## Current State
- Upgrade flow calls Flask `/upgrade-storage` without checking if node has pool space
- No cross-node migration — storage locked to provisioned node
- Delete + recreate loses all user data

## Why `zfs send/receive` over rsync

| Factor | rsync | zfs send/receive |
|--------|-------|------------------|
| Small files (millions) | Per-file stat/open overhead: 50-100 MB/s | Block-level: file count irrelevant |
| 80GB mixed workload over 10GbE | 30-60+ min | 2-5 min (best), 8-12 min (loaded) |
| Permissions/ownership | Needs -a flag, can miss xattrs | Exact block-level replica |
| Zvol support | Must copy ext4 files individually | Copies raw block device perfectly |
| Atomicity | No — partial state possible | Snapshot = point-in-time freeze |

`zfs send` operates at block level — it doesn't care about file count, file sizes, or filesystem type. For zvols, it sends the raw ext4 block device as-is. For datasets, it sends the ZFS filesystem blocks. Both produce exact replicas.

## Design

### Decision Logic (backend upgrade handler)
```
1. Calculate additionalGb = newQuotaGb - currentQuotaGb
2. Check current node's pool space via GET /host-space
3. If currentNode has (additionalGb + headroom) --> IN-PLACE UPGRADE
4. Else --> scan other nodes via selectStorageNode(newQuotaGb)
5. If another node found --> CROSS-NODE MIGRATION
6. If no node found --> FAIL "insufficient space across all nodes"
```

### Cross-Node Migration Sequence
```
Backend (NestJS)              Source Node Flask            Target Node Flask
================              ================            ================

1. LOCK USER
   status='migrating'
   (blocks sessions + file ops)

2. POST /migrate-snapshot ---->
                              zfs snapshot
                              datapool/users/{uid}@migrate
                              <-- {snapshotSize, name}

3. POST /migrate-send ------->
                              zfs send -c ...@migrate |
                                ssh target zfs receive
                                datapool/users/{uid}
                              <-- {ok, bytesSent, durationSec}

4. POST /migrate-finalize -------------------------------->
                                                          Resize: volsize/quota to newQuotaGb
                                                          For zvol: resize2fs
                                                          Mount /datapool/users/{uid}
                                                          NVMe-oF target setup (if zvol)
                                                          chown 1000:1000
                                                          Verify mount + permissions
                                                          <-- {ok, usedBytes, mountOk}

5. DB TRANSACTION:
   UPDATE user_storage_volumes
     SET nodeId=new, quota_bytes=new
   INSERT storage_extensions

6. POST /deprovision -------->
                              NVMe-oF teardown, unmount
                              zfs destroy -f (incl snapshot)
                              <-- {ok}

7. UNLOCK USER
   status='active'
```

### Rollback Strategy (CRITICAL)
```
Failure at step 2 (snapshot):  --> Unlock user. Source untouched.
Failure at step 3 (send):     --> Destroy snapshot on source.
                                   zfs destroy -f on target (partial receive).
                                   Unlock user.
Failure at step 4 (finalize): --> Deprovision on target (full cleanup).
                                   Destroy snapshot on source.
                                   Unlock user.
Failure at step 5 (DB):       --> Deprovision on target.
                                   Destroy snapshot on source.
                                   Unlock user.
Failure at step 6 (cleanup):  --> Log warning. Migration already succeeded.
                                   Source cleanup becomes async/manual.
                                   User is on new node.
```

Source data is NEVER deleted until migration is verified AND DB is updated.

### Concurrency Controls
- **Migration semaphore**: Max 2 concurrent migrations per node (as source or target)
- **Volume lock**: `status='migrating'` blocks all user operations
- **Backend queue**: If slots full, return 429 "Migration in progress, try again later"
- **Timeout**: `max(300s, usedDataGb * 30s)` — scales with data size
- **I/O priority**: `ionice -c2 -n7` on zfs send to avoid starving active sessions

### Additional Considerations
- **User sessions must be stopped** before migration (existing check already blocks active sessions)
- **Billing continuity**: charges continue on old volume until DB swap, then new rate applies
- **SSH key auth between nodes**: required for `zfs send | ssh target zfs receive`
- **Storage network preferred**: use `ipStorage` (10GbE) for send, fall back to `ipCompute`
- **Snapshot cleanup**: always destroy `@migrate` snapshot in finally block

## Task 1: Flask — Migration endpoints in app.py

**File**: `host-services/storage-provision/app.py`

### 1a. `/migrate-snapshot` (POST) — SOURCE node
Creates a ZFS snapshot for consistent point-in-time transfer.
```python
# Request: { "storageUid": "u_xxx" }
# Executes: sudo zfs snapshot datapool/users/{uid}@migrate
# Response: { "ok": true, "snapshotName": "...@migrate", "snapshotSizeBytes": 12345 }
```
Also returns `usedBytes` (from `df` for zvol, `zfs get used` for dataset) for verification later.

### 1b. `/migrate-send` (POST) — SOURCE node
Streams snapshot to target via SSH pipe over 10GbE.
```python
# Request: { "storageUid": "u_xxx", "targetIp": "10.10.100.88" }
# Executes:
#   sudo zfs send -c datapool/users/{uid}@migrate | \
#     ssh -o StrictHostKeyChecking=no zenith@{targetIp} \
#     sudo zfs receive -F datapool/users/{uid}
# Response: { "ok": true, "bytesSent": ..., "durationSec": ... }
```
- `-c` flag: compressed send (reduces network bytes for sparse zvols)
- `-F` flag on receive: force overwrite if partial exists from failed prior attempt
- Uses `ionice -c2 -n7` to lower I/O priority
- SSH cipher: `aes128-gcm@openssh.com` for speed (hardware-accelerated)
- Timeout: `max(300, usedDataGb * 30)` seconds

### 1c. `/migrate-finalize` (POST) — TARGET node
Sets up the received dataset/zvol for use on target.
```python
# Request: { "storageUid": "u_xxx", "newQuotaGb": 10, "storageBackend": "zfs_zvol" }
# Steps:
#   For zvol: zfs set volsize=10G, resize2fs, mount, fstab, NVMe-oF target, chown
#   For dataset: zfs set quota=10G, verify mount, chown
# Response: { "ok": true, "mountOk": true, "usedBytes": ..., "permissionsOk": true }
```

### 1d. `/migrate-cleanup` (POST) — SOURCE or TARGET
Cleans up after failure. Idempotent — safe to call multiple times.
```python
# Request: { "storageUid": "u_xxx", "action": "destroy_snapshot" | "destroy_all" }
# "destroy_snapshot": just removes @migrate snapshot
# "destroy_all": full deprovision (NVMe-oF teardown + unmount + zfs destroy)
```

## Task 2: Backend — Enhanced upgrade with migration orchestration

**Files**: `backend-new/src/storage/storage.service.ts`, `backend-new/src/storage/storage.controller.ts`

### 2a. `checkNodeSpace(nodeId, requiredGb)` in storage.service.ts
Queries specific node's available space.
```typescript
async checkNodeSpace(nodeId: string, requiredGb: number): Promise<{
  hasSpace: boolean; availableGb: number; headroomGb: number;
}>
```

### 2b. `migrateStorage(...)` in storage.service.ts
Full migration orchestrator with rollback:
```typescript
async migrateStorage(
  storageUid: string,
  sourceNode: Node,
  targetNode: Node,
  newQuotaGb: number,
  storageBackend: string,
): Promise<{ ok: boolean; error?: string; bytesSent?: number; durationSec?: number }>
```

Implementation:
1. Call `/migrate-snapshot` on source → get snapshot info
2. Call `/migrate-send` on source with target's `ipStorage || ipCompute`
3. Call `/migrate-finalize` on target with newQuotaGb
4. Return success — caller handles DB + source cleanup

Rollback wrapper: try/catch around each step, cleanup on any failure.

### 2c. Enhanced PATCH handler in storage.controller.ts

```
1. [EXISTING] Verify ownership, check no active sessions
2. [NEW] Fetch volume's node from DB
3. [NEW] Check current node space for additionalGb + headroom
4. [BRANCH A - IN-PLACE] Has space:
     --> Existing /upgrade-storage flow
     --> DB update: quota_bytes + storage_extensions
5. [BRANCH B - MIGRATE] No space:
     --> Set status = 'migrating' (locks volume)
     --> selectStorageNode(newQuotaGb) to find target
     --> If none: revert status, return 400 "No space"
     --> Call migrateStorage(...)
     --> On success: DB TRANSACTION {
           UPDATE nodeId, quota_bytes, storageBackend
           INSERT storage_extensions
           SET status = 'active'
         }
     --> Deprovision source (best-effort, log if fails)
     --> On failure: revert status to 'active', return 500
6. [EXISTING] Audit log, return response
```

## Prerequisites

- **SSH key auth between nodes**: `ssh-copy-id zenith@{targetIp}` on each node pair for passwordless `zfs send | ssh ... zfs receive`. Must use the storage network IP.
- **sudo for zfs commands**: Flask uses `sudo zfs send/receive/snapshot/destroy`. zenith user must have NOPASSWD sudo for these.
- **10GbE connectivity**: Nodes must reach each other via `ipStorage` (10.10.100.x) for fast transfer.

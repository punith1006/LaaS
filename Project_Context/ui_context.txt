OAuth/Keycloak Flow
User clicks OAuth provider button on /signin or /signup
Frontend stores idpHint in session storage
Redirects to Keycloak authorization endpoint
User authenticates with provider (institution SSO or external IdP)
Keycloak redirects back to /callback with authorization code
Frontend exchanges code → POST /api/auth/oauth/callback with { code, redirectUri, idpHint }
Backend exchanges code with Keycloak, returns tokens
Frontend saves tokens and redirects to /home

Proactive: App-shell polls every 30s and refreshes if token expires in <2 min
Reactive: On any API 401, refreshes once and retries request
Deduplication: Mutex prevents concurrent refresh calls
Protected Routes

If no token, redirects to /signin

8. KEY ARCHITECTURAL DECISIONS
Authentication
Token Storage: LocalStorage (accessible to XSS, but no sensitive data in JWT payload)
Strategy: Proactive refresh before expiration + reactive on 401 (belt-and-suspenders)
Deduplication: Mutex pattern prevents thundering herd on concurrent 401s
API Communication
Centralized Wrapper: apiFetch() handles all auth logic, retry logic, and error mapping
Error Handling: Maps array error messages from backend, shows user-friendly toasts
Mock Mode: Fallback mock data when API_BASE not configured (dev-friendly)
Routing
Protected Routes: Layout-level auth check (console layout) + route-level redirects
Root Routing: Server-side cookie check in root page.tsx (better UX, no client-side redirect flash)
Grouped Layouts: Auth routes use (auth) and console routes use (console) for layout isolation
State Management
Minimal Zustand: Only used for signup flow (cross-step persistence)
Component State: React hooks for UI state (modals, tabs, forms)
No Redux/Context: Lightweight approach suitable for current scope
Styling
CSS Variables Over Tailwind: Core colors/sizing via CSS custom properties for dynamic theming
No CSS-in-JS: Pure CSS + Tailwind (faster, no runtime overhead)
Responsive: Mobile-first (mobile-optimized auth layout with desktop enhancements)

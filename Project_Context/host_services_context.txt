A. Session Orchestration Service (Port 9998)
Purpose: Manages the complete Docker container lifecycle for GPU desktop sessions.Key Endpoints:
Endpoint	Method	Purpose	Auth
/health	GET	Returns Docker, GPU, Selkies image, MPS daemon status	None
/sessions/launch	POST	Async session launch (returns immediately with launch ID)	X-Session-Secret
/sessions/<name>/events	GET	Get real-time launch events and status	X-Session-Secret
/sessions/<name>/status	GET	Get Docker container state	X-Session-Secret
/sessions/<name>/stop	POST	Gracefully stop session (30s timeout)	X-Session-Secret
/sessions/<name>/restart	POST	Restart container	X-Session-Secret
/sessions/<name>/logs	GET	Get last 100 lines of container logs	X-Session-Secret
/sessions	GET	List all active LaaS session containers	X-Session-Secret
/resources	GET	Get CPU cores, ports, displays allocation status	X-Session-Secret
Resource Allocation Strategy:
CPU Cores: Allocatable range 2-15 (cores 0-1 reserved for OS); contiguous allocation only
Ports: Nginx range 8100-8199; derived ports (Selkies +1000, metrics +11000)
Displays: X display range 20-99
Reserved Infrastructure Ports: 3000 (Grafana), 3001 (Uptime Kuma), 3100 (Loki), 8080 (cAdvisor internal), 8999 (cAdvisor published), 9090 (Prometheus), 9093 (Alertmanager), 9100 (Node Exporter), 9115 (Blackbox), 9400 (DCGM), 9500 (MPS), 9501 (Session Exporter), 9998 (Session Orch), 8180 (Keycloak)
Session Launch Workflow (9 Steps):
Scheduling - Validate launch parameters (session_id UUID, tier, vCPU 1-14, memory, VRAM, SM%)
Allocating Ports - Find first available nginx port (8100-8199), display number, Selkies/metrics derived ports
Allocating CPUs - Find contiguous block of requested cores from 2-15
Validating NFS Mount - For stateful sessions only; verify /mnt/nfs/users/<storage_uid> exists and is writable
Creating Container - Build comprehensive Docker command with:
GPU access (--gpus all)
Resource limits (CPU cores, memory, pids-limit)
Security (seccomp, capabilities, AppArmor)
Environment variables (CUDA, HAMi, MPS, TURN, Selkies)
Volume mounts (MPS pipes, HAMi libs, sudoers, bash configs, lxcfs, storage)
Docker labels for monitoring (session_id, user_id, tier, node)
Starting Container - Launch with 30s timeout
Waiting for Desktop - Poll nginx port (2s intervals, 120s max) until HTTP 200 response
Health Checking - Verify WebRTC stream accessibility
Ready - Mark session ready with connection info (ports, password, URL, hostname)
Tier Slugs: spark, blaze, inferno, supernovaNetworking Modes:
Bridge Mode (Isolated): Containers on laas-sessions network; HTTP ports published to host
Host Mode (Legacy POC): Full host network access (deprecated; security risk)

B. Storage Provision Service (Port 9999)
Purpose: Manages ZFS user storage provisioning, deprovisioning, and file operations.Key Endpoints:
Endpoint	Method	Purpose	Auth
/health	GET	Check service and ZFS pool availability	None
/provision	POST	Create ZFS dataset with quota for user	X-Provision-Secret
/deprovision	POST	Destroy ZFS dataset (idempotent)	X-Provision-Secret
/host-space	GET	Get datapool total/available space	None
/upgrade-storage	POST	Increase storage quota safely	X-Provision-Secret
/storage/usage/<storage_uid>	GET	Get used bytes vs quota	X-Provision-Secret
/files/<storage_uid>	GET	List files in user storage	X-Provision-Secret
/files/<storage_uid>/mkdir	POST	Create folder	X-Provision-Secret
/files/<storage_uid>/upload	POST	Upload files (multipart)	X-Provision-Secret
/files/<storage_uid>/download	GET	Download file	X-Provision-Secret
/files/<storage_uid>/delete	DELETE	Delete file/folder recursively	X-Provision-Secret
Storage Architecture:
ZFS Pool: datapool (e.g., RTX 5090 node with large NVMe)
Dataset Structure: datapool/users/<storage_uid>
Storage UID Format: u_[0-9a-f]{24} (52 chars total)
Default Quota: 5GB (configurable 1-50GB)
Mount Point: /mnt/nfs/users/<storage_uid>
Container Mount: /home/ubuntu (bind mount inside container)
Provisioning Flow:
Pre-check: Verify datapool and datapool/users dataset exist
Run provision script: sudo /usr/local/bin/provision-user-storage.sh <storage_uid> <quota_gb>
Post-verify: Check dataset exists with correct quota
Optional NFS setup (if ENABLE_NFS_AUTOMOUNT=true):
Add export line to /etc/exports
Create mount at /mnt/nfs/users/<storage_uid>
Add /etc/fstab entry
Deprovisioning Flow (Idempotent):
Validate storage_uid format
If NFS automount enabled: unmount, remove exports/fstab
Destroy ZFS dataset with -f flag (force unmount + snapshots)
Verify destruction
Remove mountpoint directory
Quota Upgrade Strategy:
Uses zfs set quota=<new_gb>G (instantaneous, safe)
Validates new quota > current quota
Verifies upgrade succeeded; rolls back on failure
ZFS Size Parser:
Accepts formats: 24K, 5.23G, 1234567890 (bytes)
Multipliers: B=1, K=1024, M=1024², G=1024³, T/P/E follow
File Operations Security:
Directory traversal prevention (rejects .., normalizes paths)
Quota enforcement (OS/ZFS prevents writes beyond quota)
Secure filename handling (werkzeug.utils.secure_filename)
PermissionError handling for filesystem issues
2. Host Configuration Files
Location: c:\Users\Punith\LaaS\host-services\config\
bash.bashrc
Injected into all containers; provides:
Electron app sandbox disable (for VS Code, Chrome, Slack)
LD_PRELOAD setup for resource interceptors (fake_sysconf.so, libvgpu.so)
HAMi VRAM/SM enforcement via LD_PRELOAD wrapping (Python, NVCC, Jupyter)
Smart sudo wrapper (strips LD_PRELOAD to prevent system tool crashes)
Package manager wrappers (apt, pip) that strip LD_PRELOAD
nvidia-smi-wrapper
Shells out to real /usr/bin/nvidia-smi.real; intercepts output to:
Show per-container VRAM limit instead of full GPU memory
Reads limit from CUDA_DEVICE_MEMORY_LIMIT_0 or /tmp/cudevshr.cache
Replaces VRAM display: sed "s|/ [0-9]*MiB|/ ${LIMIT_MIB}MiB|g"
passwd-wrapper
Strips LD_PRELOAD environment to prevent HAMi conflicts during password change
seccomp-gpu-desktop.json
Seccomp filter (default ALLOW) that blocks dangerous syscalls:
Kernel module operations (kexec_load, init_module, delete_module)
BPF (bpf), process inspection (process_vm_writev/readv)
Keying (add_key, keyctl, request_key)
Namespace operations (unshare, setns)
System operations (reboot, swapon, mount, umount2, pivot_root)
Returns ERRNO 1 on attempt
sudoers
Base sudoers file (overrides base image); includes @includedir /etc/sudoers.d
sudoers-laas-user
Comprehensive sudo policy for ubuntu user:
Grant: Full sudo (NOPASSWD: ALL)
Deny: Module loading, filesystem operations, networking, containerization, code interpretation
Defense in depth: Denies both symlink AND real binary paths
Environment: Clears LD_PRELOAD and LD_LIBRARY_PATH automatically
supervisord-hami.conf
Process supervisor config managing all container services:
entrypoint (priority 1) - Desktop environment startup
dbus (priority 1) - D-Bus system daemon
selkies-gstreamer (priority 20) - WebRTC encoder (if not using KasmVNC)
kasmvnc (priority 20) - Alternative VNC encoder
nginx (priority 30) - Reverse proxy for WebRTC
pipewire-group (priority 10) - Audio daemon group (pipewire, wireplumber, pipewire-pulse)

Key Risks & Caveats
Port Conflicts: The reserved ports list in session orchestration is hardcoded; changes require code modification
Storage Quota: Default 5GB is not enforced at the API level; relies on ZFS quota mechanism
NFS Automount Fragile: Single-host POC uses file-backed ZFS; production must validate with native pools
Monitoring Gaps: Custom exporters (mps-exporter, session-exporter) are critical but not standardized
Bridge Networking Migration: POC uses host networking; bridge mode migration requires careful testing
HAMi Compliance: Resource limits enforced at CUDA level; CPU limits depend on docker --cpuset-cpus
Session Secret Leakage: X-Session-Secret transmitted in headers (use TLS in production)
lxcfs Dependency: System-wide tool; failure affects all containers silently
Recommendations
Immediate: Add health checks for lxcfs and nvidia-patch status to monitoring stack
Documentation: Create runbook for nvidia-patch re-application after driver updates
Testing: Validate bridge networking mode thoroughly before production
Security: Enforce TLS 1.3+ for all HTTP APIs (both on host and backend)
Scaling: Plan orchestrator integration with Prometheus file_sd discovery
Capacity: Monitor Docker overlay partition monthly; add automated cleanup cron
Alerting: Implement MPS fault frequency trending (3+ faults/hour threshold)

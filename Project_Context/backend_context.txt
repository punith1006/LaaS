5. Key Business Logic
Session Management:
Resource-based allocation: vCPU, memory, GPU VRAM tracked per session
Wallet holds: Pre-authorized amount before session launch
Status transitions: pending → starting → running → (reconnecting) → stopping → ended
Idle timeout, spend limit enforcement, node failure handling
Reconnection tracking with exponential backoff (Session.reconnectCount)
Session orchestration via HTTP API (100.100.66.101:9998 default)
Billing System:
Compute: Duration × rate_per_hour (tracked in BillingCharge)
Storage: Quota_GB × price_per_GB_per_month (hourly processing at 00:00)
Spend limits: Daily/monthly/date_range with 85% warning + 1-hour runway warning
Wallet holds: Estimated cost pre-authorized before session launch
Immutable records: BillingCharge and WalletTransaction tables never deleted (audit trail)
Storage Lifecycle:
5GB free SSO allocation (sso_default type, never charged)
Purchased extensions: Payment-linked StorageExtension records
OS switching: Creates OsSwitchHistory, may migrate old volume to new
Status workflow: provisioning → active → (wiping) → wiped
ZFS datasets + NFS exports managed on external host
GPU Fractional Sharing:
HAMI (Hardware-Managed isolation): SM percentage configurable per config
GPU VRAM allocation: Exclusive or shared modes
Node has total GPU VRAM; sessions reserve allocated amount
Actual vs. config VRAM tracked for accuracy
Credit Validation:
Pre-session: Wallet balance >= estimated session cost
During session: Periodic spend limit checks
Post-session: Final billing charge created, wallet balance debited
Frozen wallet: Prevents new session launches

Key Architectural Decisions
Immutable Audit Tables: BillingCharge, WalletTransaction, AuditLog, NodeResourceReservation never deleted (compliance)
Denormalized Costs: Session.cumulativeCostCents cached for fast queries
Resource Snapshots: Allocation snapshot stored at launch to prevent config changes mid-session affecting billing
Serializable Transactions: Session launch uses isolation level to prevent double-allocation
Fastify over Express: Better performance for high-throughput file uploads (500MB limit)
External Session Orchestration: HTTP API calls to separate service (not in-process)
Wallet Holds Pattern: Pre-authorized amount prevents overdraft

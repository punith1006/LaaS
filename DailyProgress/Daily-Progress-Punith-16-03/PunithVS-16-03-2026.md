## 1. Database schema and application data model

- **Relational schema finalized**  
  - Defined core entities for the LaaS platform: **Users**, **Organizations**, **Roles**, **Projects/Labs**, and **Storage metadata** (including per-user `storage_uid` and provisioning status).
  - Normalized relationships so that:
    - A **User** belongs to exactly one **Organization** but can hold multiple **Roles** (e.g. `student`, `instructor`, `admin`).
    - Storage-related fields (`storage_uid`, `storageProvisioningStatus`, `storageProvisioningError`) are tracked on the user record for auditability.
  - Ensured all foreign keys, uniqueness constraints, and indexes are in place to support:
    - Fast lookup on login by `email` or external identity (`keycloak_user_id` / OAuth subject).
    - Efficient querying of users by organization and role for admin/analytics views.
- **Schema safety and migrations**  
  - Captured all changes as repeatable migrations (auth columns, storage fields, indexes).
  - Applied and validated migrations on a clean database and on an existing dev database to confirm smooth upgrades.

## 2. Login, registration, OAuth, and SSO flows

- **Unified authentication model**  
  - Implemented a single backend auth pipeline that supports:
    - **Local accounts** (email + password, mainly for internal/testing).
    - **Public OAuth** providers (Google/GitHub) with standard OAuth 2.0 / OIDC claims.
    - **Institution SSO** via Keycloak (LaaS Academy realm), treated as a first-class SSO flow.
  - Introduced an explicit `authType` field (`local`, `public_oauth`, `university_sso`) used:
    - In the database for analytics and authorization rules.
    - In JWTs and `/me` responses so the frontend can adapt based on how the user authenticated.
- **User registration and first-login behavior**  
  - Implemented **just-in-time** user creation on first successful OAuth/SSO login:
    - Creates a user record, derives organization membership from the identity provider, and attaches appropriate roles.
    - On subsequent logins, reuses the existing user by external ID and updates profile fields as needed.
  - For **institution SSO users**, user creation also kicks off the storage provisioning workflow (15GB ZFS quota) in the background; failures are logged and surfaced in the UI without blocking login.
- **Session and token handling**  
  - Standardized on backend-issued JWTs (signed with `JWT_SECRET`) regardless of upstream IdP:
    - Validate upstream tokens/claims once, then issue a short-lived API JWT plus refresh token.
    - Ensure a consistent claim structure (`sub`, `authType`, `orgId`, `roles`) across all flows.
  - Hardened token validation logic (expiry, signature, audience) and added structured logging for authentication events.

## 4. Remote access / streaming: from Guacamole to WebRTC

- **Initial implementation: Apache Guacamole**  
  - Set up Apache Guacamole as the first remote desktop/streaming layer for containers and remote desktops.
  - Verified that it works well for **basic workflows** (terminal access, light GUI apps, general development tasks) and integrated it with the auth model.
  - During evaluation, identified clear limitations for **latency-sensitive scenarios**:
    - Interactive **video editing**.
    - Real-time **Blender** rendering and complex 3D workflows.
    - In these cases, RDP/VNC-based streaming introduced noticeable input and rendering lag.
- **Decision and redesign: move to WebRTC**  
  - Based on profiling and user experience testing, concluded that the Guacamole-based stack was not sufficient for high-fidelity, low-latency media workloads.
  - Led a redesign of the streaming architecture around **WebRTC**, emphasizing:
    - End-to-end **low latency** over UDP.
    - High frame rate and resolution support suitable for media and 3D use cases.
    - Fine-grained control over encoder parameters and adaptive bitrate.
- **Implementation and migration considerations**  
  - Evaluated signaling approaches (REST + WebSocket) to coordinate session setup between browser clients and container-side agents.
  - Defined a media pipeline that exposes a dedicated WebRTC endpoint per lab session, with proper authentication and authorization tied back to LaaS users.
  - Treated this as a major change, iterating through multiple test cycles to validate stability, quality, and backward compatibility with existing lab workflows.
- **Current status**  
  - The **WebRTC-based streaming path is now in place** and integrated with the LaaS platform:
    - Latency is effectively negligible for real-time editing and 3D scenarios in internal tests.
    - Session establishment, teardown, and reconnection are robust.
    - From the user perspective, the remote desktop for latency-sensitive labs feels effectively local.
  - Apache Guacamole remains available for simpler, non-latency-critical access patterns, but the primary path for production-grade media and 3D workloads is WebRTC.

HAMi-Core Build and Validation
What HAMi-Core Does
HAMi-core provides VRAM and SM (compute) limits per container via LD_PRELOAD interception of CUDA API calls. It is NOT a kernel module or daemon — it is a shared library (libvgpu.so) that intercepts cuMemAlloc, cuMemGetInfo, and related calls.

Key mechanism: Each container that loads libvgpu.so via LD_PRELOAD creates or attaches to a file-backed shared memory region at /tmp/cudevshr.cache. This file stores the memory limits and per-process usage tracking. Limits are read from environment variables at init time.

Step 1 — Build libvgpu.so
sudo apt install -y build-essential cmake git
git clone https://github.com/Project-HAMi/HAMi-core.git
cd HAMi-core && mkdir build && cd build
cmake .. -DCMAKE_BUILD_TYPE=Release
make -j$(nproc)
ls -la libvgpu.so   # Confirm: ~418784 bytes

# CORRECT install path (DEVIATION from original plan):
sudo cp libvgpu.so /usr/lib/libvgpu.so   # NOT /usr/local/lib/
sudo ldconfig
ls -la /usr/lib/libvgpu.so
⚠ CRITICAL DEVIATION: libvgpu.so must be at /usr/lib/libvgpu.so — NOT /usr/local/lib/. This is the correct path for bare metal Ubuntu. Using the wrong path causes silent failures where HAMi initializes but LD_PRELOAD is never resolved.

Step 2 — CUDA MPS Daemon
# Stop any manually-started MPS first:
echo 'quit' | sudo nvidia-cuda-mps-control 2>/dev/null || true

sudo tee /etc/systemd/system/cuda-mps.service > /dev/null << 'EOF'
[Unit]
Description=CUDA MPS Control Daemon
After=nvidia-persistenced.service
[Service]
Type=forking
Environment=CUDA_VISIBLE_DEVICES=0
Environment=CUDA_MPS_PIPE_DIRECTORY=/tmp/nvidia-mps
Environment=CUDA_MPS_LOG_DIRECTORY=/tmp/nvidia-log
ExecStartPre=/usr/bin/nvidia-smi -pm 1
ExecStart=/usr/bin/nvidia-cuda-mps-control -d
ExecStop=/bin/bash -c "echo quit | /usr/bin/nvidia-cuda-mps-control"
Restart=on-failure
RestartSec=10
[Install]
WantedBy=multi-user.target

sudo systemctl daemon-reload
sudo systemctl enable --now cuda-mps
sudo systemctl status cuda-mps   # Expect: active (running)
⚠ NOTE: MPS get_server_list returning empty is NORMAL. Servers spawn on-demand when a CUDA process connects. This is not a failure.

Step 3 — HAMi Enforcement Validation
This is the definitive test. Run inside a container with LD_PRELOAD set:
# Install torch inside container:
pip install torch --index-url https://download.pytorch.org/whl/cu128 -q --break-system-packages

# Run enforcement test:
python3 -c "
import torch
total = torch.cuda.get_device_properties(0).total_memory
print(f'Reported VRAM: {total/1024**3:.1f}GB')   # Should show 4.0GB

print('Trying to allocate 6GB (should FAIL if limit is 4GB)...')
try:
    t = torch.zeros(int(6*1024**3/4), dtype=torch.float32, device='cuda')
    print('FAIL: limit NOT enforced')
except RuntimeError as e:
    print(f'SUCCESS: blocked — {e}')

print('Trying to allocate 3GB (should SUCCEED)...')
try:
    t = torch.zeros(int(3*1024**3/4), dtype=torch.float32, device='cuda')
    print('SUCCESS: 3GB allocated fine')
except RuntimeError as e:
    print(f'FAIL: blocked at 3GB — unexpected')
"


IMPORTANT — nvidia-smi always shows full GPU: nvidia-smi queries NVML directly at the kernel driver level. HAMi cannot intercept NVML calls — only CUDA API calls. This means nvidia-smi will always report the full GPU VRAM regardless of limits. This is expected behavior, not a bug. The actual enforcement (HAMi OOM) is real and works correctly. The recommended approach: wrap the nvidia-smi binary (see Section 7) and document this limitation in Terms of Service.

Technical detail — how HAMi stores limits: When a CUDA process with LD_PRELOAD loads, HAMi creates or attaches to /tmp/cudevshr.cache — a file-backed mmap'd shared region (struct shared_region_t, ~2MB). The limit[] array at offset ~1600 bytes holds per-device VRAM limits in bytes. The first process to create the file writes the limits from env vars. Subsequent processes in the same container attach to the same file and verify consistency. Each container gets its own isolated cache file (different Docker overlay filesystem). This is per-container and correct.

bash.bashrc Injection for User Shell Sessions
User terminal sessions (Konsole inside the desktop) are non-login shells. /etc/profile.d/ scripts are NOT sourced. LD_PRELOAD must be injected via /etc/bash.bashrc:
# Get original bashrc from image
docker run --rm --entrypoint cat \
  ghcr.io/selkies-project/nvidia-egl-desktop:latest \
  /etc/bash.bashrc > /tmp/bash.bashrc.orig

# Append HAMi injection
cat >> /tmp/bash.bashrc.orig << 'EOF'

# HAMi CUDA interception — applied to all bash sessions
# NOT set globally — only activates when user opens a terminal
# VGL/X/KDE startup is unaffected (uses /etc/entrypoint.sh, not login shells)
if [ -f /usr/lib/libvgpu.so ] && [ "${HAMI_INJECTED}" != "1" ]; then
  export LD_PRELOAD=/usr/lib/libvgpu.so
  export HAMI_INJECTED=1
  mkdir -p /tmp/vgpulock 2>/dev/null
fi
EOF
chmod 644 /tmp/bash.bashrc.orig
⚠ DEVIATION: profile.d scripts are NOT sourced for non-login shells. Konsole opens non-login interactive shells. /etc/bash.bashrc is the correct injection point. This was discovered after multiple failed attempts with profile.d.

Binary Wrappers
nvidia-smi Wrapper
nvidia-smi segfaults when both libvgpu.so LD_PRELOAD and NVML are active simultaneously. Wrap it to strip LD_PRELOAD before execution:
cat > /tmp/nvidia-smi-wrapper << 'EOF'
#!/bin/bash
# Strip HAMi LD_PRELOAD before calling real nvidia-smi
exec env -u LD_PRELOAD /usr/bin/nvidia-smi.real "$@"
EOF
chmod +x /tmp/nvidia-smi-wrapper
🔵 MVP BUILDER NOTE: nvidia-smi will always show the full host GPU VRAM (e.g., 24GB for RTX 4090, 32GB for RTX 5090) regardless of container limits. This is a kernel driver limitation — NVML bypasses all user-space interception. The wrapper prevents segfaults. Do NOT attempt to spoof nvidia-smi output — focus on real enforcement (HAMi) and document the limitation transparently.

passwd Wrapper
cat > /tmp/passwd-wrapper << 'EOF'
#!/bin/bash
# Strip LD_PRELOAD before calling real passwd (prevents segfault)
exec env -u LD_PRELOAD /usr/bin/passwd.real "$@"
EOF
chmod +x /tmp/passwd-wrapper

NFS Persistent Storage Setup
Architecture
Each user gets exactly 15GB of persistent storage as a ZFS dataset on the NAS, mounted via NFS to whichever node their session runs on. This is the source of truth for user data across all sessions.


POC Setup Commands
sudo apt install -y zfsutils-linux nfs-kernel-server

# POC: file-backed pool (no dedicated partition available)
sudo mkdir -p /vg_containers
sudo truncate -s 20G /vg_containers/nas_pool.img
sudo zpool create -f datapool /vg_containers/nas_pool.img

# Create user volumes
sudo zfs create datapool/users
sudo zfs create -o quota=5G datapool/users/testuser1
sudo zfs create -o quota=5G datapool/users/testuser2

# NFS export (loopback for POC)
echo '/datapool/users 127.0.0.1(rw,sync,no_subtree_check,no_root_squash)' | sudo tee -a /etc/exports
sudo exportfs -ra
sudo systemctl restart nfs-kernel-server

# Mount
sudo mkdir -p /mnt/nfs/users
sudo mount -t nfs4 127.0.0.1:/datapool/users /mnt/nfs/users

# Create home dirs with correct ownership (uid 1000 = ubuntu inside container)
for i in 1 2 3 4; do
  sudo mkdir -p /mnt/nfs/users/testuser$i
  sudo chown 1000:1000 /mnt/nfs/users/testuser$i
done
⚠ DEVIATION: POC uses 20GB file-backed pool with 5GB quotas (not 15GB) due to root partition space constraints. Production uses dedicated NAS hardware with 15GB quotas per user as specified in the infra plan.
🔵 MVP BUILDER NOTE: The NFS path exposed inside containers is 127.0.0.1:/datapool/users/testuser1 — this leaks internal infrastructure details. In production, use a resolvable internal hostname (e.g., nas.internal) instead of 127.0.0.1, and use abstract user IDs as path components instead of usernames (e.g., /homes/u_a7f3c2). The orchestrator must provision a new ZFS dataset and NFS export when a new user registers, and mount it to /home/ubuntu in every session that user launches.

Resource Visibility: lxcfs
Problem
Without intervention, containers show the host's full hardware specs in /proc and KDE System Settings. A user on a 4-core/8GB tier sees the full 32-core/64GB host machine. This exposes internal infrastructure and is a trust/security issue.

Solution: lxcfs
lxcfs is a FUSE filesystem that intercepts /proc reads and returns container-aware values matching Docker cgroup limits.
# Install and start on HOST (not inside containers):
sudo apt install -y lxcfs
sudo systemctl enable --now lxcfs

# Verify it provides all needed files:
ls /var/lib/lxcfs/proc/
# Expected: cpuinfo  diskstats  loadavg  meminfo  slabinfo  stat  swaps  uptime


CPU NOTE: lxcfs alone does not fix nproc. --cpus=4 limits CPU time (throttling) but all 32 cores remain visible. Add --cpuset-cpus=0-3 to docker run to restrict visibility to 4 physical cores. lxcfs then correctly reports 4 in /proc/cpuinfo.
🔵 MVP BUILDER NOTE: df -h will always show the container overlay device and NFS mount path. This is a kernel-level reality — containers see their actual mounts. Mitigation for production: (1) Use abstract NFS paths with no identifying information. (2) The overlay root line (Device: overlay) is normal and reveals nothing sensitive. (3) Do not expose an SSH shell to users — the desktop-only UX limits what df output can expose.

Validated Container Launch Parameters
Complete Working docker run Command
This is the exact validated command for launching one Selkies desktop session. All deviations and required flags are documented inline:
# Prepare per-container vgpulock dir (MUST be per-container, not shared):
sudo mkdir -p /tmp/vgpulock-$i && sudo chmod 777 /tmp/vgpulock-$i

docker run -d \
  --name selkies-$SESSION_ID \

  # Resource limits (adjust per tier):
  --gpus all                          # GPU access
  --cpus=4                            # CPU time limit
  --cpuset-cpus=0-3                   # CPU visibility (lxcfs needs this for nproc)
  --memory=8g                         # RAM hard limit

  # IPC and networking:
  --ipc=host                          # Required for Selkies shared memory (WebRTC)
  --network=host                      # POC ONLY — required for WebRTC over Tailscale
  --tmpfs /dev/shm:rw                 # Shared memory for WebRTC

  # Display and streaming:
  -e DISPLAY=:$DISPLAY_NUM            # Unique per container (e.g., :20, :21, :22, :23)
  -e DISPLAY_SIZEW=1920 -e DISPLAY_SIZEH=1080 -e DISPLAY_REFRESH=60 \
  -e SELKIES_ENCODER=nvh264enc        # Hardware NVENC encoding

  # Auth:
  -e SELKIES_ENABLE_BASIC_AUTH=true \
  -e SELKIES_BASIC_AUTH_PASSWORD=$TOKEN \  # Per-session random token
  -e PASSWD=mypasswd                  # Desktop login password

  # Port assignment (unique per container — --network=host requires this):
  -e NGINX_PORT=$NGINX_P              # Web UI port (e.g., 8081, 8082, 8083, 8084)
  -e SELKIES_PORT=$SELKIES_P          # Internal gstreamer port (e.g., 9081, 9082)
  -e SELKIES_METRICS_HTTP_PORT=$METRICS_P \

  # GPU:
  -e CUDA_VISIBLE_DEVICES=0           # GPU index
  -e CUDA_NVRTC_ARCH=89               # sm_89 for Ada (sm_100 for Blackwell RTX 5090)
  -e __NV_PRIME_RENDER_OFFLOAD=1 \
  -e __GLX_VENDOR_LIBRARY_NAME=nvidia \

  # HAMi VRAM limits (adjust per tier):
  -e CUDA_DEVICE_MEMORY_LIMIT_0=4096m    # 4GB VRAM limit for this container
  -e CUDA_DEVICE_SM_LIMIT=25             # 25% SM (compute) limit
  -e CUDA_MPS_PINNED_DEVICE_MEM_LIMIT='0=4G' \
  -e CUDA_MPS_ACTIVE_THREAD_PERCENTAGE=25 \

  # TURN server for WebRTC (required for remote access through NAT/firewall):
  -e SELKIES_TURN_HOST=$TURN_HOST \
  -e SELKIES_TURN_PORT=3478 \
  -e SELKIES_TURN_USERNAME=$TURN_USER \
  -e SELKIES_TURN_PASSWORD=$TURN_PASS \
  -e SELKIES_TURN_PROTOCOL=tcp \

  # Volume mounts:
  -v /mnt/nfs/users/$USER_ID:/home/ubuntu \  # User persistent storage
  -v /usr/lib/libvgpu.so:/usr/lib/libvgpu.so \  # HAMi library
  -v /tmp/vgpulock-$i:/tmp/vgpulock \          # Per-container HAMi lock dir
  -v /usr/bin/nvidia-smi:/usr/bin/nvidia-smi.real \  # Real nvidia-smi
  -v /tmp/nvidia-smi-wrapper:/usr/bin/nvidia-smi \   # Wrapper replaces it
  -v /usr/bin/passwd:/usr/bin/passwd.real \
  -v /tmp/passwd-wrapper:/usr/bin/passwd \
  -v /tmp/supervisord-hami.conf:/etc/supervisord.conf \  # Critical custom config
  -v /tmp/bash.bashrc.orig:/etc/bash.bashrc \           # HAMi injection for terminals

  # lxcfs /proc spoofing:
  -v /var/lib/lxcfs/proc/cpuinfo:/proc/cpuinfo:ro \
  -v /var/lib/lxcfs/proc/meminfo:/proc/meminfo:ro \
  -v /var/lib/lxcfs/proc/stat:/proc/stat:ro \
  -v /var/lib/lxcfs/proc/uptime:/proc/uptime:ro \
  -v /var/lib/lxcfs/proc/loadavg:/proc/loadavg:ro \
  -v /var/lib/lxcfs/proc/diskstats:/proc/diskstats:ro \
  -v /var/lib/lxcfs/proc/swaps:/proc/swaps:ro \

  ghcr.io/selkies-project/nvidia-egl-desktop:latest

⚠ PRODUCTION NETWORK CHANGE: --network=host is used in the POC because Tailscale VPN is the remote access method and WebRTC requires direct port access. In production, Cloudflare Tunnel handles remote access — it terminates at the host and reverse-proxies to containers. This means --network=bridge can and SHOULD be used in production for proper network namespace isolation between containers. With --network=bridge, use -p $NGINX_P:8080 and configure the Cloudflare Tunnel to proxy to the container's exposed port.

Multi-Container Port Assignment
The Port Problem and Solution
With --network=host, all containers share the host network namespace. Multiple containers cannot all bind to port 8080. Each container must be assigned unique ports via environment variables.

⚠ DEVIATION: The original plan used -p 8080:8080 port mapping. This causes 'Connection failed' in WebRTC when using --network=host. With --network=host, -p flags are ignored. Use SELKIES_PORT and NGINX_PORT env vars only. In production with --network=bridge, revert to -p $NGINX_P:$INTERNAL_PORT.
🔵 MVP BUILDER NOTE: The orchestrator port registry: maintain a mapping of (node_id, slot_number) → port assignments. When launching a container, assign the next available slot on the target node. On container stop, free the slot. With --network=bridge in production, port conflicts are eliminated — each container has its own network namespace.

Session Lifecycle for Orchestrator
🔵 MVP BUILDER NOTE: This section defines exactly what the orchestrator must do at each stage of a user session. These are the infrastructure operations — the software layer calls these.

Session Launch Sequence
Check resource availability on target node (VRAM budget, CPU slots, RAM headroom)
Provision NFS home dir if first session for this user: zfs create -o quota=15G datapool/users/$uid
Create per-container vgpulock dir: mkdir -p /tmp/vgpulock-$session_id && chmod 777
Generate session token: TOKEN=$(openssl rand -hex 16)
Assign port slot (NGINX_PORT, SELKIES_PORT, METRICS_PORT, DISPLAY number)
Execute docker run with all parameters from Section 9
Poll container health: wait for nginx to listen on NGINX_PORT (check every 2s, timeout 60s)
Return session URL and token to frontend when ready (~15-25 seconds)

Session Health Check
# Check if nginx is up (container ready for connection):
ss -tlnp | grep :$NGINX_PORT

# Or via docker inspect:
docker inspect --format='{{.State.Status}}' selkies-$SESSION_ID

# Check selkies-gstreamer is running (streaming ready):
docker exec selkies-$SESSION_ID grep 'RUNNING' /tmp/selkies-gstreamer-entrypoint.log | tail -1

Session Termination
Send graceful warning to user (15-min, 5-min before scheduled end)
docker stop selkies-$SESSION_ID  (SIGTERM → graceful KDE shutdown → 30s timeout → SIGKILL)
docker rm selkies-$SESSION_ID  (destroys writable overlay — all system-level changes lost)
User NFS home dir is preserved (persists indefinitely)
Free port slot in orchestrator registry
Free VRAM budget on node
Record session end, duration, resource usage for billing

Session Reconnection
Container continues running if browser closes mid-session (until scheduled end time). The orchestrator returns the same NGINX_PORT and token to the user when they reconnect — WebRTC re-establishes automatically.

Container Startup Time
Current measured cold-start time: ~15-25 seconds to KDE Plasma fully visible in browser. Primary bottleneck is KDE Plasma initialization. Future optimization: pre-warm idle containers per node so users get instant sessions.

The D3cold GPU reset bug (Proxmox forum, NVIDIA forums) — RTX 5090 specific. Must test VM passthrough stability if Architecture B is ever used.

Security Gaps to Address Before Production
--network=host in POC: Switch to --network=bridge in production with Cloudflare Tunnel. This is the most critical security fix — --network=host means all containers share the host network namespace.
--ipc=host: Required for Selkies WebRTC shared memory. Evaluate whether --ipc=private breaks WebRTC in production environment.
df -h leaks host partition info: Use abstract NFS paths (hostname not IP, obfuscated user IDs).
nvidia-smi always shows full GPU: Document in ToS. Cannot be fixed at kernel level.
Kata Containers (optional upgrade path): Provides genuine VM-level isolation (own kernel, own /proc) while keeping Docker API. Requires KVM validation on Proxmox nodes. Adds ~2-3 seconds to startup. Investigate for v2.

sudo tee /usr/local/bin/calc-sm-percent << 'EOF'
#!/bin/bash
# Usage: calc-sm-percent <container_vram_gb> <total_gpu_vram_gb>
# Returns: integer SM% to use for CUDA_MPS_ACTIVE_THREAD_PERCENTAGE
#
# RTX 4090 POC:  total_gpu_vram_gb = 24
# RTX 5090 PROD: total_gpu_vram_gb = 32

CONTAINER_VRAM=${1:?Usage: calc-sm-percent <container_vram_gb> <total_gpu_vram_gb>}
TOTAL_VRAM=${2:?Usage: calc-sm-percent <container_vram_gb> <total_gpu_vram_gb>}

# Integer math: floor(vram/total * 100), minimum 5%
PCT=$(( (CONTAINER_VRAM * 100) / TOTAL_VRAM ))
[ $PCT -lt 5 ] && PCT=5

echo $PCT
EOF
chmod +x /usr/local/bin/calc-sm-percent

What You Need to Know | Details
How to launch a session | docker run with the exact parameters in Section 9. No Kubernetes, no Swarm — direct Docker API.
How to check if session is ready | Poll for nginx listening on NGINX_PORT. Typically ready in 15-25 seconds.
How to stop a session | docker stop → docker rm. NFS home persists. Everything else is ephemeral.
How to provision user storage | zfs create -o quota=15G datapool/users/$uid on NAS. Mount via NFS at container launch.
How VRAM limits work | env vars at docker run time → HAMi reads them at first CUDA call → writes to /tmp/cudevshr.cache. Limits are real and enforced.
Why nvidia-smi shows full GPU | NVML kernel bypass. Cannot be fixed. Document in ToS. Real enforcement is HAMi.
Port assignment | NGINX_PORT=8080+slot, SELKIES_PORT=9080+slot, DISPLAY=:19+slot. Unique per container on same node.
Multi-node session routing | NFS home follows user to any node. Orchestrator picks node with available VRAM budget. Return correct host IP + port to frontend.
lxcfs requirement | Must be installed and running on every host node. Required for RAM/CPU visibility in containers.
Pre-built files needed on host | supervisord-hami.conf, bash.bashrc.orig, nvidia-smi-wrapper, passwd-wrapper, libvgpu.so. These must be deployed to /tmp/ on each node before any sessions launch.
Session startup latency | ~15-25 seconds cold start. Pre-warming pool recommended for production UX.
The Selkies image OS | Ubuntu 24.04 internally (not 22.04). Software installs inside container use apt for Ubuntu 24.04 packages.

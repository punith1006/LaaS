## THE USER MODEL — 

### Account Structure
- Every user (university member) will **one individual account** with a unique identity (e.g., `priya@cs.university.ac.in`)
- Semester/department/role is a **Keycloak group** (authorization label), NOT a login credential. i am not sure of keycloak whether the universities have it or not.
- Shared logins are explicitly rejected — individual accounts are non-negotiable
- Users authenticate via **university SSO** (Keycloak federated with university Google Workspace or LDAP) — same credentials as their college portal
- for public users (who are not associated members of university and want temporary compute, its should have oauth or sso with google (for emphermeal only interactions)

### OS Choice (Chosen Once at Registration) - for the first time when the user is building there stateful system
- Ubuntu 22.04 LTS for now! and i ll provide other distros later
- This is their environment for their entire membership
- They can switch OS at any time, BUT this **permanently wipes their entire 15GB personal storage** — requires explicit typed confirmation ("DELETE") as consent
- There is no dual-OS — one user, one OS (one personal computer), one persistent environment

### Storage — Final Model
- Each user gets exactly **15GB of personal persistent storage** — their "personal layer"
- This is an LVM thin volume (`lv_user_<uid>`) on VG_STATEFUL
- It contains: their home directory (`/home/<uid>/` on Linux), their saved files, their personal settings, their project work
- **The base VM image is NOT counted in their 15GB** — it is a separate read-only shared layer
- 15GB is a hard limit enforced at the LVM volume level — cannot be exceeded
- Storage persists across ALL sessions, ALL compute configs, INDEFINITELY (for their membership tenure)
- Storage is OS-specific: if they switch OS, the old 15GB is wiped and a new 15GB begins fresh

### Base VM Images (Shared, Read-Only, Maintained by Platform)
- **One base images exist** on the platform: (this is an estimate or an example)
  - `lv_base_linux` (~60GB): Ubuntu 22.04 LTS + MATLAB, Python, CUDA toolkit, Blender, development tools
- These are **read-only** — no user can modify them
- They are **replicated on all 4 nodes**
- A software update to the base image = every user gets it on their next session automatically
- Users cannot install system-level software (no admin rights in VM); only user-profile-level changes are saved to their 15GB layer

### Session Mechanics -> (think this through critically, and check if there are better ways to do it or if this is the best approach)
- When a session launches: a **thin LVM clone** of the base image is created for that session + user's 15GB volume is mounted into the VM
- When a session ends: the thin clone of the base image is **destroyed** (any system-level changes gone — clean slate)
- The user's 15GB personal volume: **always preserved**
- This means: same software environment every session (base image), personal files always there (15GB layer)

## TWO ENGAGEMENT MODES

### Mode 1 — Stateful GUI Desktop (Persistent, CPU-Only)
- Full remote desktop experience (Linux GUI via software rendering + xrdp/Guacamole in browser) — Linux based distro as OS for now!
- CPU-only VMs (no GPU) — sufficient for MATLAB computation, Python coding, office work, light CAD, development tools
- Users who need GPU must use Tier 1 (Full Machine, exclusive) or Tier 3 ephemeral GPU containers
- User's 15GB personal layer persists across all sessions
- Multiple users can run stateful GUI sessions simultaneously on the same physical machine (each in their own isolated KVM VM)
- Hardware-level isolation (AMD-V): absolute process, memory, and data isolation between concurrent VMs
- Users also have CLI access within their stateful environment

### Mode 2 — Ephemeral Compute Sessions (Temporary)
- Jupyter Notebook, Code-Server (VS Code in browser), or SSH CLI
- Session data (scratch) is wiped after session ends
- The ephemeral instance will have pre-defined capacity (based on the config chosen by the user) — so their notebooks and code files are persistent even though the session itself is ephemeral
- for the public users, their work like the jupyter file will be saved in their account. (like how google colab operates), they can essentially spin a compute for this jupyter notebook if they want to.!!
- Available as CPU-only or with GPU time-slicing (via Docker containers using CUDA MPS + HAMi-core for software-enforced VRAM limits). GPU sharing in containers has weaker isolation than VMs — acceptable for temporary/ephemeral workloads but documented to users.
- concurrent users per machine sharing resources (depending on what config they choose) (depending on the available resources in the machine / fleet and the configs available) 
- No booking required for ephemeral — launch immediately.

## THREE-TIER COMPUTE ARCHITECTURE

### Tier 1 — Full Machine (Exclusive, GPU Included)
- One user gets the entire node exclusively: all CPU cores, RAM, and the full RTX 5090 GPU (32GB VRAM via vfio-pci passthrough to a KVM VM)
- Maximum 4 concurrent Tier 1 users across the fleet (one per node)
- For: GPU-accelerated workloads — deep learning, 3D GPU rendering, CUDA development, heavy simulation
- GPU is ONLY available in this tier for stateful sessions (hardware limitation: consumer GPUs cannot be partitioned across VMs)

### Tier 2 — Multi-User Stateful GUI Desktops (Concurrent, CPU-Only)
- Multiple KVM VMs per node, no GPU access
- Full GUI desktop experience via software rendering (xrdp + Guacamole in browser)
- All proprietary software available and functional in CPU computation mode (MATLAB, Python, Blender CPU rendering, development tools)
- Hardware-level isolation between concurrent VMs (AMD-V) — strongest possible isolation
- For: coding, MATLAB computation, document work, data analysis, light CAD, general development

### Tier 3 — Ephemeral Compute (Concurrent, Multi-User), subjected to user choice
- Docker containers for Jupyter, Code-Server, SSH CLI
- Available as CPU-only or with GPU time-slicing (GPU shared via CUDA MPS + HAMi-core in containers — software-enforced VRAM and compute limits, weaker isolation than VMs, acceptable for ephemeral/temporary workloads)
- For: quick compute runs, notebook execution, ML training/inference, public user access

## STORAGE ARCHITECTURE
The disk storage in each host machine will be hard split into handle stateful machines and ephemeral instances (for absolute data isolation and ensure no empheral instance ruins of corrupts the data at the stateful side as it would have user personal data, proprietory software and configs)
 
### Per-Node NVMe Layout (2TB per machine)
```
nvme0n1p1  128GB  EXT4   → OS + swap
nvme0n1p2  900GB  LVM PV → VG_STATEFUL (base images + active thin clone deltas)
nvme0n1p3  700GB  LVM PV → VG_EPHEMERAL (container scratch, temp, wiped on session end)
nvme0n1p4  240GB  EXT4   → Shared read-only datasets
```

### Centralized NAS (5th machine — required for multi-user scale)
- TrueNAS Scale or Synology with ZFS, 4×4TB HDDs (~12TB usable RAIDZ1)
- Hosts ALL user 15GB personal volumes (NFS over 15GbE to all 4 nodes)
- Benefit: any user can run on any available node — their files follow them
- Node failure: user reconnects on a different node, same files
- Backup target: daily ZFS snapshots

### Base Images on Each Node (Replicated) (for now the idea (for stateful machines) is to create one base VM image and then create thin 
- `lv_base_linux` ~60GB — replicated on all 4 nodes
- Active thin clone deltas: ~20GB per active session (local NVMe, fast)
- Destroyed when session ends

## USER EXPERIENCE — KEY FLOWS

### Onboarding (once)
1. User registers on `lab.youragency.in` with university email
2. Chooses OS: (permanent until they explicitly switch)
3. Account Registered: account created → 15GB LVM volume provisioned on NAS
4. Logs in with university SSO (same credentials as their college portal)

### Booking and Launching a Session
1. Login → Dashboard shows storage usage, upcoming bookings, fleet status
2. Click "Book Session" → select compute config → select time (now or scheduled)
3. At session time: "Launch" button appears → click → ~15–20 seconds → desktop appears in browser
4. Work. Save files to personal home directory. Files persist on NAS.
5. 15-min and 5-min warnings sent before session end
6. Graceful shutdown → thin clone destroyed → resources freed → billing recorded

### Session Reconnection
- If browser closes mid-session: VM continues running (till the subscribed time)
- User reopens portal → "Reconnect" button → resumes exactly where they left off

### OS Switch (with consent)
- Account Settings → Switch OS → explicit "DELETE" confirmation → all 15GB wiped → fresh environment

1. What changes when it’s a real college (IIT‑Delhi)?
There are three sides:

IIT‑Delhi – runs its own IdP (SAML or OIDC) and owns student accounts.
Your Keycloak (laas realm) – acts as a broker between IIT‑Delhi and your LaaS app.
Your LaaS app + DB – consumes Keycloak tokens and maps them to users, universities, university_idp_configs, etc.
A. One-time setup with IIT‑Delhi
Agree on protocol

Decide with IIT‑Delhi IT: SAML 2.0 or OIDC (both supported by Keycloak).
Decide which claims/attributes they will send: at minimum email, plus name, id.
Exchange metadata/config

If SAML:
IIT‑Delhi gives you: IdP metadata URL/XML (EntityID, SSO URL, certificate).
You give them: Keycloak SAML SP metadata for the laas realm.
If OIDC:
IIT‑Delhi gives you: issuer URL, authorize/token/userinfo endpoints, JWKS URL, client ID/secret (if they’re the OP).
Or you give them your Keycloak endpoints if Keycloak is acting as OP.
Configure IdP in laas realm

In Keycloak admin, realm laas → Identity Providers → Add:
Type: SAML or OpenID Connect.
Alias: iit-delhi (this is the idpAlias).
Fill in the IIT‑Delhi metadata (SAML) or issuer/discovery URL & client credentials (OIDC).
Configure mappers so IIT‑Delhi’s attributes map to Keycloak standard ones:
email → email
given_name / first_name
family_name / last_name
sub / uid → external_id (not stored directly but used to link).
Register IIT‑Delhi in your DB

In universities (or universities-like table): create a row:
name = "IIT Delhi", short_name = "IITD", domain_suffixes = ["iitd.ac.in"], etc.
In university_idp_configs:
university_id → IIT‑Delhi row.
idp_type = "saml" or "oidc".
keycloak_idp_alias = "iit-delhi".
Other IdP metadata as needed.
In frontend institutions config (or API), add:
{ id: "iit-delhi", name: "IIT Delhi", slug: "iit-delhi", domain: "iitd.ac.in", idpAlias: "iit-delhi" }.
At this point, IIT‑Delhi is a real institution in your system, wired via Keycloak.

Marked as “federated from IdP iit-delhi”.
Keycloak then issues its own tokens for client laas-frontend (OpenID Connect):
access_token (RS256 signed, with sub, email, etc.).
id_token, refresh_token (if enabled).
Keycloak redirects back to your LaaS app

Browser is redirected to your app’s redirect_uri, e.g.:
https://laas.yourdomain.com/callback?code=XYZ....
This hits your Next.js /callback route.
Backend exchanges code for LaaS tokens

Your frontend /callback sends code + redirectUri to your NestJS backend (/api/auth/oauth/callback).
Backend’s KeycloakService.exchangeCode talks to Keycloak:
Exchanges code for tokens (access_token, refresh_token).
Calls userinfo to get profile (email, name, sub).
In AuthService.handleOAuthCallback:
Normalises email (punith.vs@gmail.com).
Looks for an existing LaaS user with keycloakSub = <kc-sub> or email = punith.vs@gmail.com.

Creates a user in your users table:
email = "punith.vs@gmail.com".
firstName = "Punith", lastName = "VS".
authType = "oauth", oauthProvider = "keycloak" / "iit-delhi".
defaultOrgId set to the IIT‑Delhi organization/university row.

Records login in loginHistory (method = oauth, provider = IIT‑Delhi).

Frontend stores LaaS tokens and logs Punith in

/callback receives the tokens from backend and calls saveTokens().
Shows toast “Signed in successfully” and redirects to /dashboard.
From now on, calls to your backend use your JWT in Authorization: Bearer ....
3. Key points in this real-world IIT‑Delhi scenario
IIT‑Delhi never shares passwords with you; they only assert “this is Punith” via SAML/OIDC.
Keycloak is the broker:
For the app, it’s your IdP.
For IIT‑Delhi, it’s a service provider/client.
Your LaaS app:
Treats IIT‑Delhi users as first-class users in your DB.
Uses your own JWTs/session model, independent of IIT‑Delhi, but linked via fields like keycloakSub and university_idp_configs.keycloak_idp_alias.
So in simple words:

IIT‑Delhi logs in Punith → tells your Keycloak “this is Punith” → Keycloak tells LaaS “this is Punith” → LaaS creates/uses its own user record and tokens, while keeping IIT‑Delhi as the source of truth for his identity.

4. Understanding Keycloak Session Management (Dual-Realm Architecture)
================================================================================

Architecture Overview:
--------------------
LaaS uses a dual-realm Keycloak setup:

  laas realm (BROKER)
  ├── Has laas-frontend client (for OAuth: Google, GitHub)
  ├── Has IdP: laas-academy (for institutional SSO)
  ├── All user SESSIONS are created here
  └── User cookies: KEYCLOAK_SESSION, KEYCLOAK_IDENTITY

  laas-academy realm (INSTITUTION IdP)
  ├── Has laas-realm-broker client (for brokering from laas)
  ├── Has test users (student accounts)
  └── Only authenticates users, does NOT create sessions for laas-frontend

Key Insight:
-----------
When a user logs in via "Sign in with institution" (LaaS Academy):
  1. User authenticates against laas-academy realm
  2. laas-academy verifies credentials
  3. laas realm receives the assertion
  4. laas realm CREATES the session (in laas realm!)
  5. User cookies are set by laas realm, NOT laas-academy

Same for Google OAuth:
  1. User authenticates against Google
  2. laas realm receives the callback
  3. laas realm CREATES the session (in laas realm!)

Bottom Line:
-----------
ALL sessions are in the laas realm. The laas-academy realm only
authenticates users but does NOT maintain sessions for the laas-frontend client.

5. The "Already Authenticated" Error - Root Cause and Fix
================================================================================

The Error:
---------
"You are already authenticated as different user 'student' in this session.
Please sign out first."

Why It Happens:
--------------
1. User logs in via SSO (session created in laas realm, cookies set)
2. User signs out from the app (local tokens cleared)
3. User tries to log in via Google OAuth
4. Keycloak sees the KEYCLOAK_SESSION cookie for laas realm
5. Keycloak shows "already authenticated" because a session exists

The Fix Requires TWO Things:
---------------------------

A. Keycloak Client Configuration (Valid Post Logout Redirect URIs)
   In laas realm → Clients → laas-frontend → Access settings:
   - Add: http://localhost:3000/signin to "Valid post logout redirect URIs"
   This allows the logout endpoint to redirect back to the app.

B. Frontend Sign-Out Code Changes
   The sign-out must:
   1. Store the id_token BEFORE clearing localStorage
   2. Call Keycloak logout endpoint with id_token_hint
   3. Include post_logout_redirect_uri

Required Code Changes (app-shell.tsx):
------------------------------------
```
const performSignOut = () => {
  // Get ID token BEFORE clearing storage
  const idToken = getIdToken();
  
  // Clear local tokens
  localStorage.clear();
  sessionStorage.clear();
  
  // Redirect to Keycloak logout
  const keycloakUrl = process.env.NEXT_PUBLIC_KEYCLOAK_URL;
  const keycloakRealm = process.env.NEXT_PUBLIC_KEYCLOAK_REALM || "laas";
  const appUrl = window.location.origin;
  
  if (keycloakUrl) {
    let logoutUrl = `${keycloakUrl}/realms/${keycloakRealm}/protocol/openid-connect/logout`;
    const params = new URLSearchParams();
    
    if (idToken) {
      params.set("id_token_hint", idToken);
    }
    params.set("post_logout_redirect_uri", `${appUrl}/signin`);
    logoutUrl += `?${params.toString()}`;
    
    window.location.href = logoutUrl;
  }
};
```

Required Token Storage (token.ts):
-------------------------------
```
export function saveTokens(tokens: AuthTokens & { idToken?: string }): void {
  localStorage.setItem(ACCESS_KEY, tokens.accessToken);
  localStorage.setItem(REFRESH_KEY, tokens.refreshToken);
  if (tokens.idToken) {
    localStorage.setItem(ID_TOKEN_KEY, tokens.idToken);
  }
}

export function getIdToken(): string | null {
  return localStorage.getItem(ID_TOKEN_KEY);
}
```

Required Backend Change (auth.service.ts):
---------------------------------------
In handleOAuthCallback(), return id_token from Keycloak:
```
const kcTokens = await this.keycloak.exchangeCode(code, redirectUri);
const idToken = kcTokens.id_token;  // Store this!
return {
  ...tokens,
  idToken,  // Include in response
};
```

Why id_token_hint is Required:
----------------------------
- Keycloak logout endpoint REQUIRES id_token_hint to identify which session to terminate
- Without it, Keycloak returns "Missing parameters: id_token_hint"
- The id_token is stored in the user's browser during login
- It's retrieved during logout to terminate the correct session

The Flow After Fix:
-----------------
1. User logs in via SSO
2. Backend returns id_token, frontend stores it in localStorage
3. User signs out
4. Frontend retrieves id_token, calls Keycloak logout with id_token_hint
5. Keycloak terminates the session, clears cookies
6. User logs in via Google OAuth - works because session is gone

6. OAuth Buttons - prompt=login Parameter
================================================================================

Even with proper logout, OAuth buttons should include prompt=login:

```
function getOAuthUrl(provider: "google" | "github"): string {
  const params = new URLSearchParams({
    client_id: KEYCLOAK_CLIENT_ID,
    redirect_uri: callbackUrl,
    response_type: "code",
    scope: "openid email profile",
    kc_idp_hint: provider,
    prompt: "login",  // Forces fresh authentication
  });
  params.append("_", Date.now().toString());  // Cache bust
  return `${KEYCLOAK_URL}/realms/${KEYCLOAK_REALM}/protocol/openid-connect/auth?${params}`;
}
```

What prompt=login Does:
---------------------
- Tells Keycloak to IGNORE any existing session
- Forces the user to authenticate fresh
- Acts as a safety net if logout didn't fully terminate the session
- Works for Google, GitHub, and any OAuth provider

7. Troubleshooting Keycloak Logout Issues
================================================================================

Error: "Invalid redirect uri"
-----------------------------
Cause: post_logout_redirect_uri is not configured in the client
Fix: Add the redirect URI to laas-frontend client in laas realm

Error: "Missing parameters: id_token_hint"
------------------------------------------
Cause: Logout called without id_token_hint parameter
Fix: Store id_token during login, use it in logout URL

Error: "Invalid parameter: id_token_hint"
----------------------------------------
Cause: id_token doesn't match the session (different realms)
Fix: Ensure OAuth and SSO both use the laas realm

Error: "Client not found" in laas-academy
----------------------------------------
Cause: Trying to logout from laas-academy realm directly
Fix: Always logout from laas realm where sessions are created

Still Getting "Already Authenticated" After Logout:
-------------------------------------------------
1. Clear browser cookies for localhost:8080 manually
2. Check if id_token is being stored during login
3. Verify logout URL includes id_token_hint
4. Use incognito window to test (isolated cookies)

## THE USER MODEL — 

### Account Structure
- Every user (university member) will **one individual account** with a unique identity (e.g., `priya@cs.university.ac.in`)
- Semester/department/role is a **Keycloak group** (authorization label), NOT a login credential. i am not sure of keycloak whether the universities have it or not.
- Shared logins are explicitly rejected — individual accounts are non-negotiable
- Users authenticate via **university SSO** (Keycloak federated with university Google Workspace or LDAP) — same credentials as their college portal
- for public users (who are not associated members of university and want temporary compute, its should have oauth or sso with google (for emphermeal only interactions)

### OS Choice (Chosen Once at Registration) - for the first time when the user is building there stateful system
- Ubuntu 22.04 LTS for now! and i ll provide other distros later
- This is their environment for their entire membership
- They can switch OS at any time, BUT this **permanently wipes their entire 15GB personal storage** — requires explicit typed confirmation ("DELETE") as consent
- There is no dual-OS — one user, one OS (one personal computer), one persistent environment

### Storage — Final Model
- Each user gets exactly **15GB of personal persistent storage** — their "personal layer"
- Stored as a ZFS dataset on the centralized NAS (`datapool/users/<uid>`) with a 15GB quota, mounted via NFS to whichever node their session runs on
- It contains: their home directory (`/home/<uid>/` on Linux), their saved files, their personal settings, their project work
- **The base Docker image is NOT counted in their 15GB** — it is a separate read-only shared layer
- 15GB is a hard limit enforced at the ZFS quota level — cannot be exceeded
- Storage persists across ALL sessions, ALL compute configs, INDEFINITELY (for their membership tenure)
- Storage is OS-specific: if they switch OS, the old 15GB is wiped and a new 15GB begins fresh
- Works identically across all compute configs — the NFS mount is transparent to the user

### Base Image (Shared, Read-Only, Maintained by Platform)
- **One base Docker image** for all sessions (stateful and ephemeral, all tiers): `selkies-egl-desktop:ubuntu2204-custom`
- Contains: Ubuntu 22.04 LTS + KDE Plasma desktop + MATLAB, Python, CUDA toolkit, Blender, development tools
- Uses Docker read-only image layers — **no user can modify the base software**
- Pre-pulled on all 4 nodes (via local Docker registry or direct pull)
- A software update to the base image = every user gets it on their next session automatically
- Users cannot install system-level software (no admin/root rights); only user-profile-level changes (pip, conda, npm installs to `~/`) are saved to their 15GB layer on NAS
- Single image guarantees identical software environment across all tiers — a user switching from Starter (CPU-only) to Pro (GPU) finds the exact same desktop, libraries, and tools

### Session Mechanics
- When a session launches: a Docker container is started from the read-only Selkies base image + user's 15GB NFS volume is bind-mounted at `/home/<uid>`. GPU configs get HAMi-core + MPS env vars; CPU-only configs launch without GPU access.
- When a session ends: the container is stopped and removed (writable overlay destroyed — clean slate). User's NFS home directory is always preserved.
- Same mechanism for ALL tiers (Starter through Full Machine, stateful and ephemeral): same software environment every session (base image), personal files always there (15GB layer on NAS)
- System-level changes (apt-get installs) are lost on session end. User-profile changes (pip install --user, conda, saved files) persist via NFS.

### Mode 1 — Stateful GUI Desktop (Persistent)

- Available **with or without GPU**:

## ISOLATION REQUIREMENTS (Non-Negotiable)

All isolations are enforced simultaneously. All user sessions (every tier, stateful and ephemeral) run as Docker containers with container-level (strong) isolation:

### Data Isolation — Absolute
- User data lives on NAS via NFS — separate ZFS dataset per user with hard quota. No cross-user access.
- Base image is read-only (Docker image layers). No user can modify base software.
- Docker storage and VG_CONTAINERS are on separate NVMe partitions. User persistent data lives on the NAS via NFS — ephemeral container corruption cannot touch user data or base images.

### Memory Isolation — Strong
- cgroups v2 hard memory limits enforced per container (`--memory`). Kernel-enforced, cannot be exceeded.

### Process Isolation — Strong
- All sessions: Docker containers with PID namespaces, seccomp profiles, AppArmor, no-new-privileges. Users cannot see each other's processes.
- Sufficient for authenticated educational/research users. Tradeoff vs VM-level (absolute) isolation accepted in exchange for seamless cross-tier switching and uniform user experience.

### GPU Isolation — Software-Enforced (GPU sessions only)
- VRAM allocation: dual-layer enforcement (HAMi-core at CUDA API level + CUDA MPS at driver level). Hard limit — cannot be exceeded.
- Compute (SM) partitioning: HAMi-core + MPS ACTIVE_THREAD_PERCENTAGE. Prevents one user from monopolizing GPU.
- GPU virtual address space: MPS Volta+ provides per-client isolated GPU address spaces.
- GPU L2 cache / DRAM bandwidth: shared — cannot be partitioned on consumer GPUs. Noisy neighbor possible (10-30% variability).
- GPU fatal fault: propagates to co-resident containers. MPS auto-recovers (Volta+). Watchdog service restarts affected containers in 30-60 seconds. User data safe on NAS.

### Session Isolation — Enforced

## STORAGE ARCHITECTURE
The disk storage in each host machine will be hard split into handle stateful machines and ephemeral instances (for absolute data isolation and ensure no empheral instance ruins of corrupts the data at the stateful side as it would have user personal data, proprietory software and configs)
 
### Per-Node NVMe Layout (2TB per machine)
```
nvme0n1p1  128GB  EXT4   → OS + swap
nvme0n1p2  500GB  EXT4   → /var/lib/docker (Docker images, container overlays, writeable layers)
nvme0n1p3  1100GB LVM PV → VG_CONTAINERS (container scratch, ephemeral temp, wiped on session end)
nvme0n1p4  240GB  EXT4   → Shared read-only datasets
```

Note: With all sessions as containers (no KVM VMs), the previous VG_STATEFUL partition for VM templates and thin clones is no longer needed. The space is redistributed: a dedicated partition for Docker storage (images, overlays) and a larger VG_CONTAINERS for ephemeral scratch. User persistent data lives entirely on the NAS via NFS, not on local NVMe.

### Centralized NAS (5th machine — required for multi-user scale)
- TrueNAS Scale or Synology with ZFS, 4×4TB HDDs (~12TB usable RAIDZ1)
- Hosts ALL user 15GB personal volumes (NFS over 15GbE to all 4 nodes)
- Benefit: any user can run on any available node — their files follow them
- Node failure: user reconnects on a different node, same files
- Backup target: daily ZFS snapshots

### Base Image on Each Node (Replicated)
- **Selkies Docker image** (`selkies-egl-desktop:ubuntu2204-custom`) — pre-pulled on all 4 nodes. Docker read-only image layers serve as the base for ALL sessions (stateful and ephemeral, all tiers). Container writable overlay is ephemeral — destroyed when container stops.
- Single image = single build process, single software stack, seamless cross-tier switching.
